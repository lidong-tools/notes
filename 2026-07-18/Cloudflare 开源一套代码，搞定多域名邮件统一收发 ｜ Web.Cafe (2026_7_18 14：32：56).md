# Cloudflare 开源一套代码，搞定多域名邮件统一收发

客户发邮件到 support@a.com，你用 Gmail 个人邮箱回过去——客户第一反应是什么?

"这谁啊?"

哪怕你在签名里写清楚"我是 A 产品的客服"，信任度也已经掉一截了。客户期待的就是 support@a.com 回信，干净、对称、专业，不需要任何解释。

如果你手上运营多个产品，每个都有独立域名邮箱(support@a.com 、 support@b.com 、 support@c.com ……)，这个问题就被乘以 N。每加一个产品就要再配一套发信、再切一次 from 地址，工作流稍微一乱，从错误地址回信、或者干脆从 Gmail 回信的情况就会发生，客户那头的信任就在这种小细节里慢慢漏掉。

这个问题，Cloudflare 现在用 一套开源代码、一次部署 直接解决了。

## 项目叫 agentic-inbox

仓库： cloudflare/agentic-inbox

一个程序、一个 Cloudflare 账号，同时管理任意多个域名邮箱的收发，客户发到哪个地址，你就从哪个地址原路回过去。

核心特性：

- 完整邮件客户端，通过 Email Routing 收发，带富文本编辑器、回复转发线程、搜索、附件
- 每个邮箱独立隔离，跑在自己的 Durable Object 里，有独立 SQLite 数据库，附件存 R2
- 侧边栏内置 AI agent，9 个邮件工具帮你读、搜、起草、回复
- 新邮件 AI 自动起草回复，发送前必须你点确认

5 个产品开 5 个 mailbox，各自独立配置、独立数据，但 统一在一个界面里收发，from 地址永远跟客户来信完全对得上。

## 为什么现在才能做到？Cloudflare 终于能发邮件了

之前 Cloudflare Email Routing 只能收不能发，所以多域名回信一直是个坑。这次升级成 Cloudflare Email Service，Agents Week 期间公测开放，添加域名时 SPF、DKIM、DMARC 这些发信认证全自动配好，邮件正常进收件箱不进垃圾箱。

费用方面说清楚：发信功能需要 Workers 的 5 美元/月套餐(群友们基本都已经订了)，这个套餐下 每天 1000 封邮件免费，所有产品共用这个额度，一个月 3 万封，绝大多数情况用不完；就算超了也是按量计费，不会突然爆一笔。Email Routing 收信仍然完全免费。

如果你已经在用 Workers Paid，这套多域名邮箱方案的边际成本基本是零。

## 加分项 1：内置 MCP，接 Claude Code 直接管邮件

agentic-inbox 内置 MCP server，Claude Code、Cursor 这些外部 agent 可以直接连进收件箱，起草邮件等你确认再发。

理想工作流：客户来信 → AI 自动起草 → Claude Code 里结合产品文档改措辞 → 点发送，从对应域名邮箱原路回出去。整条链路不出 Cloudflare，数据全在你自己账号里。

## 加分项 2：开源代码，AI 想怎么改怎么改

这是开源项目，直接复制到自己仓库，丢给 Claude Code 想加什么加什么。比如：

- 首页加个未读消息计数，一眼看完所有 mailbox 的待处理量
- 每收到一封新邮件，自动推 Discord 或飞书通知，重要客户来信不漏接
- 按客户邮箱域名打标签，B2B 客户和个人用户自动分流
- 周末/夜间自动回"已收到，工作日内回复"，白天再让 AI 正常起草

## 部署

README 已经写得很清楚了，跟着走就行。三个小坑提醒一下：

1. 如果你之前没开过 Cloudflare Access，先去开通 (免费的 0 元套餐就够用)，否则部署后会卡在 Access 配置那一步。
2. 多域名配置在环境变量里用逗号分割就行 ，比如 `a.com,b.com,c.com` ，不用每个域名单独部署一份。
3. 不要直接从公共仓库部署 ，Cloudflare Workers 现在对公共仓库的部署可能会有问题。正确姿势：先克隆到本地，再推到自己的 私有仓库 ，从私有仓库部署。这样后面想让 AI 改代码、加功能也直接在自己仓库里改，闭环更顺。

对运营多产品的开发者来说，这是个体验质变—— 一次部署，所有域名邮箱的收发和 AI 处理全闭环，客户那头看到的永远是干净一致、专业的发件人。
